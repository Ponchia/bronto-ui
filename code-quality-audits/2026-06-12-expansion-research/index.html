Summary

Bronto UI is already unusually strong for a solo-maintained CSS-first system: the package exports a broad leaf CSS surface, machine-readable tokens/classes, renderer themes, thin framework bindings, packed consumer examples, visual/e2e gates, dead-code checks, and local security scans. This audit's adoption-infrastructure findings have now been implemented locally: a Tailwind v4 CSS theme bridge, Svelte and Vue lifecycle adapters over existing behaviors, a packed Vue example, a packed Tailwind example, and a local Figma Variables handoff built from the existing DTCG token export. Later passes also added durable background-job state and an accessible workbench splitter, so long-running async work and real tool panes now have shared primitives instead of one-off local patterns. The local browser verification paths are now named npm scripts, so the audited evidence is repeatable without translating CI YAML by hand.

Findings

Resolved finding: Tailwind v4 support now has a CSS bridge

Claim: Bronto had Tailwind layer-order documentation, but no Tailwind v4 `@theme` export, custom variants, or packed Tailwind consumer example; the current tree now ships all three.

Impact: Tailwind v4 made CSS-first theme variables, `@custom-variant`, `@reference`, and `@source` central adoption primitives. Bronto now exposes those adoption affordances without adding a component runtime.

Remediation: the CSS-only `@ponchia/ui/tailwind` export uses `@theme inline`, documents `@reference`/`@source`, and is covered by the packed `examples/tailwind-vite` smoke.

Bronto now ships a Tailwind v4 CSS bridge before adding more components.

Scope: `@ponchia/ui` package exports, Tailwind interop docs, and packed examples as of 2026-06-12 after this pass.

Evidence state: supported.

Basis: local export/doc inventory now includes the bridge; Tailwind's current docs center CSS theme variables, source registration, and reference-only imports.

Sources: S1, S4, S5

Resolved finding: behavior adapters now cover Svelte and Vue

Claim: Bronto already shipped React, Solid, and Qwik lifecycle bindings; the current tree now adds Svelte lifecycle actions, Vue directives/plugin helpers, and a packed Vue example.

Impact: cross-framework consumers can now adopt the delegated behavior layer in the same CSS-first style without Bronto becoming a component library.

Remediation: `@ponchia/ui/svelte` actions and `@ponchia/ui/vue` directives/plugin helpers call the existing delegated behavior initializers and return cleanup; markup, state, and styling stay in consumer land.

Thin Svelte and Vue lifecycle adapters fit Bronto's architecture better than per-framework components or Web Components, and now ship locally.

Scope: behavior-layer adoption surfaces and examples, not new component markup.

Evidence state: supported.

Basis: architecture explicitly favors SSR-safe delegated behaviors and optional thin bindings; the local example matrix now includes Vue and the SvelteKit example uses the Svelte adapter.

Bronto should not chase a broad shadcn/Radix-style catalog as the next SOTA move.

Scope: roadmap prioritization and public package surface.

Basis: external catalogs show the breadth available elsewhere, while Bronto's roadmap explicitly preserves a narrower report/provenance lane.

Resolved finding: background jobs now have a persistent state row

Claim: current progress/status guidance across comparable systems covers progress indicators and status semantics, but Bronto's state lane still lacked a persistent row for imports, exports, sync runs, and report generation that continue after the initiating interaction is gone.

Impact: without a durable state object, hosts would keep inventing local progress rows or hiding long-running work in toasts, creating inconsistent async UX across report and app surfaces.

Remediation: `css/state.css` now ships `.ui-job`, state modifiers, compact layout, `--job-progress`, forced-colors handling, and reduced-motion-safe progress updates. The class recipe, generated metadata, docs, demo, changelog, roadmap, and cross-browser Playwright checks were updated with explicit host-owned polling, retry, cancellation, and queue semantics.

Bronto now covers persistent background-job status as a CSS-first state primitive instead of leaving async work hidden in transient toasts.

Scope: `@ponchia/ui/css/state.css`, generated class/type metadata, state docs, state demo, and cross-browser non-pixel tests.

Evidence state: supported.

Basis: `.ui-job` is in source and dist CSS, the class registry exposes `ui.job()`, docs and demo show determinate/progressbar usage, and Chromium/Firefox/WebKit each passed progress-width, ARIA, and forced-colors checks.

Sources: S1, S3, S8

Resolved finding: workbench panes now have an accessible splitter

Claim: Bronto's workbench lane shipped inspector, property, and selection-bar primitives, but still left accessible split panes to each app. Current ARIA references and comparable libraries converge on a focusable separator with keyboard resizing and value semantics.

Impact: without a shared splitter, every real editor/workbench surface would improvise pane layout, pointer drag, keyboard resize, and `aria-valuenow` handling.

Remediation: `css/workbench.css` now ships `.ui-splitter`, pane and handle parts, vertical/horizontal modifiers, and documented sizing knobs. `initSplitter()` wires `data-bronto-splitter` hosts with SSR-safe keyboard and pointer resizing, ARIA value sync, cleanup, and `bronto:splitter:resize`; React, Solid, Qwik, Svelte, and Vue adapters expose the matching lifecycle wrapper.

Bronto now covers accessible workbench split panes without becoming a drag-and-drop or layout-persistence framework.

Scope: `@ponchia/ui/css/workbench.css`, `@ponchia/ui/behaviors`, generated class/type metadata, workbench docs/demo, framework bindings, and cross-browser non-pixel tests.

Evidence state: supported.

Basis: `.ui-splitter` is in source and dist CSS, `initSplitter` exports through the behavior barrel and every binding adapter, docs and demos include separator ARIA, and Chromium/Firefox/WebKit each passed keyboard, pointer, and forced-colors splitter checks.
