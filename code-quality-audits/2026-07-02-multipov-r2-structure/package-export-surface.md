# Package / export surface architecture — bronto-ui structure review
**Verdict:** The package surface is strong: `exports`, `files`, generated contract docs, packed consumer smoke, publint/ATTW, and stability docs mostly form a coherent public contract. The main architectural gaps are second-order: behavior leaf exports are public but less registry-backed than CSS leaves, `behaviors/internal.js` leaks through public declaration references, and the documented root CSS side-effect import is not covered by the packed TypeScript consumer gate.

**Grade:** A- — excellent contract machinery with a few public/private and leaf-registry gaps.

## Strengths
- Root is intentionally CSS-only: `exports["."]` points `style` and `default` to `./dist/bronto.css`, while JS entrypoints are explicit subpaths (`package.json:242`, `package.json:243`, `package.json:244`, `package.json:340`, `package.json:401`).
- CSS leaf conventions are highly consistent: layered public leaves map `./css/<leaf>.css` to `./dist/css/<leaf>.css`, and raw escape hatches live under `./css/unlayered/<leaf>.css` (`package.json:252`, `package.json:295`, `package.json:297`, `package.json:339`).
- CSS exports are inversely checked from the build graph, not just existence-checked: every shippable leaf must have both layered and unlayered exports, with roll-ups explicitly excluded (`scripts/check-exports.mjs:77`, `scripts/check-exports.mjs:100`, `scripts/check-exports.mjs:111`, `scripts/check-exports.mjs:118`).
- JS typed subpaths consistently use `types` + `default` conditional exports for tokens/classes/behaviors/adapters/helpers (`package.json:340`, `package.json:401`, `package.json:406`, `package.json:502`, `package.json:510`).
- The adapters-only-index shape is structurally correct, not a gap: framework bindings are defined as thin lifecycle adapters over the behavior barrel, and the binding gate derives hook/action/directive names from `behaviors/index.js` (`docs/architecture.md:91`, `scripts/check-binding-matrix.mjs:60`, `scripts/check-binding-matrix.mjs:124`, `scripts/check-binding-matrix.mjs:175`).
- Shipped artifact checks operate at tarball level: concrete export targets must be present in `npm pack`, and packed consumers import/resolve JS, JSON, CSS/doc/font assets from the installed package (`scripts/check-pack.mjs:74`, `scripts/check-pack.mjs:84`, `scripts/check-consumer-surface.mjs:314`, `scripts/check-consumer-surface.mjs:354`).
- Version/stability contract is explicit and machine-adjacent: `docs/package-contract.md` is generated from `package.json`, while `docs/stability.md` ties 1.0 readiness to export, pack, consumer, migration, and release gates (`docs/package-contract.md:1`, `docs/package-contract.md:309`, `docs/stability.md:45`, `docs/stability.md:49`, `docs/stability.md:52`).

## Weaknesses / risks
- [P2] Behavior leaf exports are manually represented in `package.json`, but I do not see an inverse gate equivalent to CSS’s “every public behavior gets `./behaviors/<leaf>`.” The current gates own the barrel exports and docs/tests, while `check-consumer-surface` only consumes whatever `package.json` already exports (`scripts/check-behavior-matrix.mjs:165`, `scripts/check-behavior-matrix.mjs:184`, `scripts/check-consumer-surface.mjs:62`, `package.json:410`). Cost: a future behavior can become public via `@ponchia/ui/behaviors` but accidentally lack the promised per-leaf import path.
- [P2] `behaviors/internal.js` is declared non-public, but its types are part of the public declaration graph: `behaviors/index.d.ts` exports `Cleanup`/`DelegateOpts` via `import("./internal.js")`, every behavior leaf declaration references it, and React/Solid/Qwik generics default to `../behaviors/internal.js` (`behaviors/internal.js:1`, `behaviors/index.d.ts:22`, `behaviors/dialog.d.ts:16`, `react/index.d.ts:8`, `solid/index.d.ts:65`). Cost: editors and API auditors see an “internal” module as part of the public type contract.
- [P2] The documented `import '@ponchia/ui'` root CSS side-effect path is not covered by packed TypeScript consumer smoke: docs recommend it in TS contexts, but `check-consumer-types` only includes export objects with `types`, excluding the root CSS export (`docs/getting-started/react-solid.md:182`, `package.json:243`, `scripts/check-consumer-types.mjs:49`). Cost: a TS framework consumer can hit root-import resolution friction that the contract gates would not catch.
- [P3] `files` ships broad public-source directories, so the tarball boundary is broader than the export boundary; `behaviors/internal.js` and declaration maps ship because `behaviors` ships wholesale (`package.json:44`, `package.json:51`, `behaviors/internal.d.ts:1`, `scripts/check-pack.mjs:38`). Cost: compliant tooling is protected by `exports`, but tarball readers and less strict tooling still see implementation artifacts.
- [P3] Behavior per-leaf exports are listed in `docs/package-contract.md` with generic “Package subpath / Public” classification, while `docs/stability.md` only names `@ponchia/ui/behaviors` as a behavior-function surface (`docs/package-contract.md:183`, `docs/package-contract.md:204`, `docs/stability.md:70`, `docs/stability.md:71`). Cost: consumers cannot tell whether `@ponchia/ui/behaviors/dialog` has the same stability/deprecation promise as the barrel.
- [P3] ATTW is present but run with `no-resolution` ignored, so it is not the final authority for root/data/CSS resolution; the custom packed consumer gates carry that responsibility instead (`package.json:188`, `package.json:189`, `scripts/check-consumer-types.mjs:4`). Cost: future maintainers may overestimate ATTW coverage unless the custom gates keep expanding with the surface.

## Top recommendations
1. Add a behavior-leaf registry/check: derive expected `./behaviors/<leaf>` exports from behavior modules or the behavior matrix, exclude `internal.js`, and fail if `package.json` misses or adds a stale behavior leaf.
2. Move public behavior shared types out of `behaviors/internal.js` into a public `behaviors/types.js` or define them directly in each public declaration path, then keep `internal.js` value-only.
3. Extend `check-consumer-types` with a tiny packed TS fixture that imports `@ponchia/ui` as a side effect under the documented framework/bundler settings, or document that TS users should prefer CSS `@import` when type resolution is strict.
4. Upgrade `gen-package-contract.mjs` classification for `./behaviors/*` from generic “Package subpath” to “Behavior leaf JS,” with the same stability language as the behavior barrel.
5. Decide whether broad `files` directories are policy. If yes, document that `exports` is the public boundary and tarball internals may ship; if no, narrow `files` or add a shipped-but-unexported allowlist.
6. Add a short “Export family conventions” table to `docs/package-contract.md`: root CSS, CSS leaf, CSS unlayered, JS barrel, JS leaf, adapter, JSON/data, schema, docs, fonts.

## Notable observations
- The behaviors-vs-adapters asymmetry is justified: behaviors are primitive modules with per-leaf imports; adapters are framework-specific aggregators whose export sets are required to mirror the behavior lifecycle surface (`scripts/check-binding-matrix.mjs:25`, `scripts/check-binding-matrix.mjs:62`, `scripts/check-binding-matrix.mjs:129`).
- CSS has the most mature contract loop in the package: source graph, generated layered outputs, layered/unlayered export pairs, component ownership, and report routing all reinforce the same surface (`scripts/build-dist.mjs:117`, `scripts/check-exports.mjs:100`, `scripts/check-component-matrix.mjs:1`, `docs/reporting.md:83`).
- The root/package mental model is clean for CSS-aware consumers but intentionally hostile to runtime JS root imports; this is well documented and encoded in the manifest (`README.md:217`, `docs/architecture.md:105`, `docs/package-contract.md:28`).
