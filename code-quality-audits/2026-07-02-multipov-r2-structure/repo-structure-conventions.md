# Repository structure, conventions & navigability — bronto-ui structure review
**Verdict:** `bronto-ui` has a strong, intentional structure: the root-as-public-subpath design is documented, generated artifacts are named and gated, and examples/docs have clear entrypoints (`docs/architecture.md:134`, `docs/architecture.md:136`, `docs/package-contract.md:301`, `examples/README.md:3`). The main newcomer friction is not chaos; it is diffusion: adding one new primitive requires collecting rules from `CONTRIBUTING.md`, `docs/architecture.md`, `package.json`, and checker source instead of following one local playbook (`CONTRIBUTING.md:77`, `docs/architecture.md:114`, `package.json:191`, `scripts/check-component-matrix.mjs:1`).
**Grade:** B+ — coherent architecture, but still too checker-mediated for first-time contributors.
## Strengths
- Canonical layout documentation exists and explains the repo’s unusual root-level public subpaths, generated artifacts, and no-`src/` publishing model (`docs/architecture.md:134`, `docs/architecture.md:143`, `docs/architecture.md:158`).
- Newcomer entrypoints are aligned: `AGENTS.md` points to contribution/layout docs, `CONTRIBUTING.md` points to the repository layout, and README/docs both provide “Start here” routes (`AGENTS.md:3`, `CONTRIBUTING.md:47`, `README.md:27`, `docs/README.md:8`).
- Generated-artifact ownership is unusually explicit: generated docs declare their generator, package contract lists provenance, and freshness is centralized in a registry/check pair (`docs/package-contract.md:1`, `docs/package-contract.md:303`, `scripts/lib/artifacts.mjs:1`, `scripts/check-fresh.mjs:1`).
- The examples/demo split has a defensible intent: examples are packed-tarball consumer apps, while demo pages are showcase/e2e fixtures (`examples/README.md:3`, `examples/README.md:9`, `README.md:38`, `test/e2e/demos.spec.mjs:5`).
- Surface ownership is machine-checked across docs/demo/e2e rather than left to memory (`scripts/check-component-matrix.mjs:1`, `docs/architecture.md:196`, `scripts/check-behavior-matrix.mjs:1`, `scripts/check-binding-matrix.mjs:2`).
- Root config is mostly purposeful, not random: Stylelint owns CSS, Prettier deliberately skips CSS/generated markdown-heavy assets, TypeScript configs document separate type/declaration gates, and Playwright explains screenshot policy (`.stylelintrc.json:1`, `.prettierignore:1`, `tsconfig.json:2`, `tsconfig.dts.json:2`, `playwright.config.mjs:4`).
## Weaknesses / risks
- [P0] None found in this structure-only pass; the public-root layout is deliberate and documented, not an accidental package shape (`docs/architecture.md:104`, `docs/architecture.md:158`).
- [P1] There is no single “add a primitive” playbook; surface admission, class registry, token generation, export wiring, docs/demo/e2e ownership, and regeneration commands are scattered across docs, manifest scripts, and checker source, costing newcomers multiple failed gate cycles (`docs/architecture.md:114`, `CONTRIBUTING.md:69`, `CONTRIBUTING.md:77`, `package.json:200`, `scripts/check-component-matrix.mjs:475`).
- [P2] The canonical layout map is not quite exhaustive: `CONTRIBUTING.md` says it classifies every top-level directory, but tracked audit artifacts and the top-level Tailwind bridge are only partially surfaced elsewhere, making the root look more accreted than it is (`CONTRIBUTING.md:47`, `code-quality-audits/2026-07-02-multipov/README.md:1`, `package.json:47`, `docs/package-contract.md:234`, `docs/architecture.md:156`).
- [P2] `test/` is structurally flat for its size: unit tests live at `test/*.test.mjs`, type tests are a single root `.d.ts`, Playwright specs/fixtures share `test/e2e/`, and screenshots sit one level deeper, so taxonomy is inferred from filenames and config rather than local structure (`package.json:192`, `playwright.config.mjs:24`, `test/types.test-d.ts`, `test/e2e/_report-print.fixture.html`, `test/e2e/__screenshots__/alerts-dark.png`).
- [P2] `demo/` has important conventions but no local map; the kitchen-sink/leaf-page/baseline ownership model is discoverable only through README prose and checker/test source, which raises the cost of adding or renaming a demo page (`README.md:40`, `test/e2e/demos.spec.mjs:15`, `scripts/check-component-matrix.mjs:341`, `test/e2e/visual.spec.mjs:8`).
- [P2] `package.json` is overloaded as both package manifest and task router: 88 scripts, a long aggregate `check` chain, and 195 exports are structurally guarded, but still hard for newcomers to scan (`package.json:115`, `package.json:191`, `package.json:242`, `scripts/check-chain.mjs:1`).
- [P3] Singular/plural naming is mostly consistent, but `legend` is a small exception across CSS/export vs docs/demo/test, adding avoidable search friction (`package.json:272`, `docs/README.md:50`, `scripts/check-component-matrix.mjs:204`, `demo/legends.html`, `test/e2e/legends.spec.mjs`).
## Top recommendations
1. Add `docs/adding-a-primitive.md` or a `CONTRIBUTING.md` section that sequences: choose lane, add CSS/classes/tokens, wire exports/files, add docs/demo/e2e matrix row, regenerate, run checks (`docs/architecture.md:114`, `CONTRIBUTING.md:77`, `package.json:200`, `scripts/check-component-matrix.mjs:475`).
2. Update `docs/architecture.md#repository-layout` to name `code-quality-audits/` and `tailwind.css`, or explicitly link to `docs/package-contract.md` for internal/dev-only paths (`docs/architecture.md:143`, `docs/package-contract.md:320`, `code-quality-audits/2026-07-02-multipov/README.md:16`, `package.json:248`).
3. Add short `test/README.md` and `demo/README.md` files that explain local taxonomy, naming, fixture policy, screenshot policy, and the checker files that enforce ownership (`playwright.config.mjs:24`, `CONTRIBUTING.md:146`, `test/e2e/demos.spec.mjs:5`, `test/e2e/visual.spec.mjs:17`).
4. Add a generated or curated task map for `package.json` scripts grouped by authoring, generation, checks, examples, e2e, release; keep `check:chain` as enforcement but stop making contributors read the whole one-line chain first (`package.json:115`, `package.json:191`, `scripts/check-chain.mjs:56`).
5. For path-frozen naming quirks, document aliases instead of renaming; start with a note that `legend.css` maps to `docs/legends.md` and `demo/legends.html` (`package.json:272`, `docs/README.md:50`, `demo/legends.html`).
6. Add a root `code-quality-audits/README.md` with retention and commit policy, since dated reports are tracked while adjacent scanner artifacts are ignored (`code-quality-audits/2026-07-02-multipov/README.md:31`, `.gitignore:38`, `.prettierignore:22`).
## Notable observations
- The absence of `src/` is a feature, not a smell: published subpath names are the root directory names, and moving them would break the package contract (`docs/architecture.md:137`, `docs/architecture.md:158`, `package.json:401`).
- The large `CHANGELOG.md` is justified as a shipped release record and release-note source, not just project history (`CHANGELOG.md:1`, `CONTRIBUTING.md:97`, `docs/package-contract.md:250`, `docs/architecture.md:271`).
- `docs/package-contract.md` is already close to a generated repo map; it could be reused to reduce duplication between package files, layout docs, and newcomer navigation (`docs/package-contract.md:7`, `docs/package-contract.md:224`, `docs/package-contract.md:307`).
