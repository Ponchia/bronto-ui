# Cascade-layer & token dependency graph — bronto-ui structure review
**Verdict:** The cascade-layer architecture is strong: `core.css` declares `@layer bronto` once, imports default leaves into it, and dist generation/checks enforce single-layer outputs (`css/core.css:10`, `css/core.css:12`, `scripts/check-dist.mjs:70`). The token graph is mostly source-backed and drift-checked, but not fully single-source: CSS-only print/preset tails and skin-unaware renderer projections create maintainability seams outside the canonical `tokens/index.js` model (`tokens/index.js:7`, `css/tokens.css:231`, `scripts/gen-charts.mjs:24`).
**Grade:** B+ — robust layer/build gates, with notable drift risk around skins, print tokens, and leaf-local variables (`scripts/check-exports.mjs:77`, `scripts/check-variables.mjs:39`).
## Strengths
- `@layer bronto` is declared as the default layer once in `core.css`, and every default import uses `layer(bronto)` in explicit intra-layer order (`css/core.css:10`, `css/core.css:12`).
- Dist output wraps flattened bundles and direct leaf imports in one `@layer bronto` block, and rejects nested/stray layer rules (`scripts/build-dist.mjs:53`, `scripts/build-dist.mjs:70`, `scripts/check-dist.mjs:70`).
- CSS leaf coverage is gated: orphan source leaves fail, opt-in leaves are enumerated, and package exports must expose both layered and raw escape-hatch targets (`scripts/check-dist.mjs:22`, `scripts/build-dist.mjs:83`, `scripts/check-exports.mjs:100`).
- Core token values have a real source of truth in `tokens/index.js`; `css/tokens.css`, `resolved.json`, and DTCG are generated/freshness-checked from it (`tokens/index.js:7`, `scripts/gen-tokens-css.mjs:24`, `scripts/lib/artifacts.mjs:38`).
- Color tiers are machine-policed: neutral/accent/status tiers are explicit, data-viz namespaces are reserved from core, and skins/charts stay opt-in (`scripts/check-color-policy.mjs:56`, `scripts/check-color-policy.mjs:142`, `scripts/check-skins.mjs:79`, `scripts/check-charts.mjs:114`).
- Renderer bridges resolve from the token model and reject `var()` leaks, so Mermaid/D2/Vega exports are not hand-edited color dumps (`scripts/lib/resolve-ref.mjs:17`, `scripts/check-mermaid.mjs:48`, `scripts/check-vega.mjs:73`).
## Weaknesses / risks
- [P1] Renderer palettes are not skin-aware: skins re-point live `--accent`, CSS data-viz uses `--chart-1: var(--accent)`, but JSON/renderer generators resolve only core light/dark `--accent`; diagrams/charts can drift visually under `data-bronto-skin` (`tokens/skins.js:20`, `css/dataviz.css:11`, `scripts/gen-charts.mjs:24`, `scripts/gen-mermaid.mjs:117`, `scripts/gen-vega.mjs:184`).
- [P2] `check-variables` is global, not import-graph/per-leaf aware: it builds one repo-wide `defined` set, so public direct leaves can pass while relying on another leaf’s definitions, e.g. `content.css` uses `--scroll-offset` from `base.css` (`scripts/check-variables.mjs:39`, `scripts/check-variables.mjs:50`, `css/content.css:83`, `css/base.css:34`).
- [P2] Print palette remaps duplicate token values in a preserved hand-authored tail, outside the JS token source; the same kind of remap also exists in `report.css`, so print semantics can drift from light tokens (`css/tokens.css:327`, `css/tokens.css:352`, `scripts/gen-tokens-css.mjs:120`, `css/report.css:687`).
- [P2] `@property` registration is useful but policy-local: `--value`, bullet, interval, and spotlight geometry are typed, while generic host geometry such as `--v`, gauge, dotmatrix, and report sizing remains bare; invalid host values fail inconsistently (`css/feedback.css:13`, `css/bullet.css:22`, `css/interval.css:11`, `css/spotlight.css:14`, `css/dots.css:447`).
- [P3] Tier 3 display knobs are documented and skin-set, but not part of the main token/resolved model; enforcement is a hard-coded known-set in `check-skins`, raising drift cost for future display knobs (`scripts/check-color-policy.mjs:98`, `tokens/skins.js:44`, `scripts/check-skins.mjs:41`).
- [P3] The generated contract table calls renderer JS files the “Source of truth,” while those files themselves are generated artifacts; contributors may edit the wrong side of the graph (`docs/package-contract.md:317`, `tokens/mermaid.js:1`).
## Top recommendations
1. Add a skin dimension to renderer outputs/helpers, or explicitly document renderer themes as core-light/core-dark only; generate skin-aware `charts/mermaid/vega` projections from `tokens/skins.js` if skins are expected to theme diagrams (`tokens/skins.js:37`, `scripts/gen-charts.mjs:102`).
2. Make `check-variables` import-graph aware: validate each public layered leaf/rollup against its declared dependencies, not the whole `css/` directory (`scripts/check-variables.mjs:29`, `scripts/check-exports.mjs:77`).
3. Move print token remaps into a generated `print` projection from `cssVars.light`, or add a first-class `print` block to `tokens/index.js` and generate both global/report print remaps (`css/tokens.css:348`, `tokens/index.js:98`).
4. Introduce a small custom-property registry for host-set numeric/length/color props, then generate `@property` blocks or an explicit “intentionally unregistered” allowlist (`css/feedback.css:7`, `css/bullet.css:22`, `css/dots.css:450`).
5. Correct the renderer source-of-truth row to name `scripts/gen-{mermaid,d2,vega}.mjs` plus `tokens/index.js`/`tokens/charts.js`, not generated `tokens/*.js` files (`docs/package-contract.md:317`, `scripts/gen-mermaid.mjs:8`).
6. Rename or clearly banner `css/generated.css` as hand-authored “generated-content trust” CSS, not a generated token artifact, to reduce token-graph confusion (`css/generated.css:1`, `css/generated.css:9`).
## Notable observations
- `css/generated.css` is hand-authored opt-in AI/system trust UI, despite the name, and `report-kit` imports it as a report leaf (`css/generated.css:2`, `css/report-kit.css:25`).
- DTCG intentionally emits CSS-runtime-derived tokens as `$value: null` with CSS extensions, while `resolved.json` carries concrete static colors for non-CSS renderers (`scripts/gen-dtcg.mjs:9`, `tokens/resolved.json:2`).
- The layer model deliberately gives unlayered consumer CSS precedence; media-boundary `!important` use is documented as a reduced-motion/print safeguard, not normal component specificity (`css/core.css:6`, `css/motion.css:397`).
