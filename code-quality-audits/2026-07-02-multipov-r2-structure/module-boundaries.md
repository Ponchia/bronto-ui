# Module boundaries & primitive-ownership — bronto-ui structure review
**Verdict:** The thesis is real at the package/import level: `core.css` is a closed identity bundle (`css/core.css:12`), opt-in leaves are exported separately (`package.json:267`), and the doctrine explicitly rejects chart scales/domain state (`docs/frontier-primitives.md:18`, `docs/architecture.md:129`). The weaker part is selector ownership inside files: primitives carry app-shell aliases (`css/primitives.css:163`, `css/primitives.css:759`), core prose knows about opt-in marks (`css/content.css:107`), and large toolbox leaves bundle multiple concerns (`css/workbench.css:1`, `css/report.css:300`).

**Grade:** B- — strong import/export boundaries, weaker intra-CSS ownership enforcement (`scripts/lib/css-leaves.mjs:17`, `scripts/check-component-matrix.mjs:1`).

## Strengths
- Default identity vs opt-in tooling is structurally enforced: root export points to `dist/bronto.css` (`package.json:242`), `core.css` imports only identity leaves (`css/core.css:12`), and `CORE_BUNDLE` is an explicit allowlist (`scripts/lib/css-leaves.mjs:23`).
- Opt-in CSS leaves are first-class package subpaths, not hidden internals: `dataviz`, `report`, `legend`, `marks`, `interval`, `spark`, `workbench`, `command`, `analytical`, and `report-kit` all have explicit exports (`package.json:267`).
- The analytical roll-up matches the declared architecture: it imports nine analytical leaves and excludes adjacent report/tooling/trust leaves (`css/analytical.css:15`, `docs/architecture.md:96`).
- Several primitives honor host-owned data/geometry boundaries in code comments and implementation: `spark` requires host-normalized `--v` (`css/spark.css:8`), `interval` requires host-normalized `--lo/--hi` (`css/interval.css:6`), and `figure` refuses chart rendering/scales (`css/figure.css:4`).
- Behavior modules mostly stay narrow and dependency-light: `command` owns filtering/navigation but leaves execution to the host (`behaviors/command.js:20`), and boundary tests forbid fetch/storage/routing/scale imports in analytical behaviors (`test/analytical-boundary.test.mjs:32`).
- Data-viz is well isolated from identity chrome: `dataviz.css` declares itself opt-in and “never UI chrome” (`css/dataviz.css:4`), and `check-charts` fails if `core.css` imports it (`scripts/check-charts.mjs:114`).

## Weaknesses / risks
- [P1] `css/primitives.css` leaks app-shell ownership: `.ui-app-metrics`, `.ui-app-metric*`, and `.ui-app-empty-state` are styled in primitives, not `app.css` (`css/primitives.css:163`, `css/primitives.css:185`, `css/primitives.css:759`). Cost: app-shell public API cannot be refactored or audited independently from primitive grammar.
- [P1] The boundary is enforced at import level, not selector level: `CORE_BUNDLE` only controls what `core.css` imports (`scripts/lib/css-leaves.mjs:17`), while component matrix is ownership coverage, not a cross-layer selector lint (`scripts/check-component-matrix.mjs:1`). Cost: future `.ui-report-*`, `.ui-app-*`, or tooling selectors can land in the wrong CSS file and still pass class/export gates.
- [P2] Core identity CSS references an opt-in evidence primitive: `.ui-prose mark:not(.ui-mark)` protects `marks.css` from being overridden (`css/content.css:107`). Cost: renaming or extracting `ui-mark` now has to update default identity CSS.
- [P2] `legend.css` is exported as a standalone leaf but its indexed and gradient modes assume `dataviz.css` chart tokens (`css/legend.css:104`, `css/legend.css:149`). Cost: consumers can import `legend.css` alone and get broken `--chart-2+`/ramp-dependent modes unless they know the implicit pairing.
- [P2] `css/workbench.css` is a grab-bag toolbox leaf: inspector, toolstrip, segmented buttons, property rows, selection bar, and splitter all ship together (`css/workbench.css:11`, `css/workbench.css:35`, `css/workbench.css:106`, `css/workbench.css:171`, `css/workbench.css:205`). Cost: consumers cannot take only splitter or only inspector CSS despite the behavior being individually exported.
- [P2] `css/report.css` mixes document shell with claim cards, evidence grids, action registers, comparison layout, meter rows, and print utilities (`css/report.css:300`, `css/report.css:425`, `css/report.css:498`, `css/report.css:585`, `css/report.css:622`, `css/report.css:667`). Cost: report primitives cannot evolve as small leaves; every report-adjacent concern increases the same file’s blast radius.
- [P2] `initSplitter` contradicts its host-policy thesis: docs say the host owns min/max policy (`behaviors/splitter.js:224`), but implementation supplies default min/max/value constants and applies them when ARIA is absent (`behaviors/splitter.js:5`, `behaviors/splitter.js:90`). Cost: behavior silently imposes layout policy.
- [P3] `@ponchia/ui/classes` is one flat registry for identity, report, analytical, workbench, command, and app-shell classes (`classes/index.js:12`, `classes/index.js:423`, `classes/index.js:612`, `classes/index.js:674`). Cost: typed consumers cannot import a layer-scoped class contract.

## Top recommendations
1. Move `.ui-app-*` compatibility aliases out of `css/primitives.css` into `css/app.css` or a tiny app-compat leaf, while keeping canonical `.ui-stat*` / `.ui-empty-state` in primitives (`css/primitives.css:163`, `css/app.css:264`).
2. Add a selector-boundary gate: primitives may not reference `.ui-app-*`/`.ui-report-*`/tooling selectors, core identity may not reference opt-in leaves except explicit allowlisted exceptions like `.ui-mark` (`css/content.css:107`, `scripts/lib/css-leaves.mjs:17`).
3. Split `workbench.css` into narrower leaves once API stability allows: `inspector/property`, `toolstrip/segmented-buttons`, `selectionbar`, and `splitter`, preserving `workbench.css` as a roll-up (`css/workbench.css:11`, `css/workbench.css:205`).
4. Split report sub-grammars or create internal sections with future leaf targets: claims/evidence, actions, compare, meter-row, and print utilities are separable concerns today (`css/report.css:300`, `css/report.css:493`, `css/report.css:622`).
5. Make `legend.css` dependency modes explicit: either add fallbacks for indexed/ramp helpers or document/gate that gradient/indexed legend variants require `dataviz.css` (`css/legend.css:104`, `css/analytical.css:10`).
6. Add layer-scoped class exports or generated subsets, e.g. `classes/identity`, `classes/report`, `classes/analytical`, while preserving the flat `cls` export (`classes/index.js:12`, `package.json:401`).

## Notable observations
- `report-kit.css` deliberately excludes workbench/command surfaces while importing nearly every static report primitive (`css/report-kit.css:9`, `css/report-kit.css:14`); that split is cleaner than the monolithic `report.css`.
- Crosshair has a decent JS/CSS split but uses JS-authored readout placement attributes consumed by CSS (`behaviors/crosshair.js:92`, `css/crosshair.css:31`); keep those names treated as a public contract.
- `connectors` is the cleanest “pure geometry plus visual grammar” example: CSS owns stroke/style (`css/connectors.css:13`), behavior measures DOMRects and calls pure geometry helpers (`behaviors/connectors.js:87`).
- The repo already has executable boundary tests for behavior imports and forbidden concerns (`test/analytical-boundary.test.mjs:24`, `test/analytical-boundary.test.mjs:32`); the missing counterpart is CSS selector ownership.
