# Build/generation pipeline & type-generation — bronto-ui structure review
**Verdict:** The architecture is strong on generated-artifact discipline: token/class artifacts are centralized in a pure registry and byte-compared, while heavier outputs have semantic gates (`scripts/lib/artifacts.mjs:31`, `scripts/check-fresh.mjs:13`, `scripts/check-charts.mjs:37`, `scripts/check-dist.mjs:60`). The weak point is orchestration and recipe typing: `build:artifacts`/`verify:artifacts` are still hand-maintained shell chains, and class recipe option interfaces are handwritten in `gen-dts.mjs` with only a partial runtime cross-check (`package.json:200`, `package.json:201`, `scripts/gen-dts.mjs:26`, `scripts/check-recipe-types.mjs:13`).
**Grade:** B+ — good gates, but no declared artifact DAG and recipe types are not source-derived (`package.json:200`, `scripts/gen-dts.mjs:26`).

## Strengths
- Token artifacts have a clear source: `tokens/index.js` feeds CSS, JSON, DTCG, resolved values, and Figma variables (`scripts/gen-tokens-css.mjs:2`, `scripts/gen-tokens-json.mjs:10`, `scripts/gen-dtcg.mjs:21`, `scripts/gen-resolved.mjs:149`, `scripts/gen-figma-variables.mjs:128`).
- The pure-artifact registry is a real single freshness frame for `.d.ts`, reference, package contract, class JSON, token mirrors, and VS Code data (`scripts/lib/artifacts.mjs:31`, `scripts/lib/artifacts.mjs:43`, `scripts/check-fresh.mjs:13`).
- Specialized artifacts combine regenerate-and-diff with domain invariants: charts check drift, shape, CVD distinguishability, and opt-in status; dist checks byte equality, orphan CSS leaves, stale outputs, layer shape, budgets, and asset URLs (`scripts/check-charts.mjs:37`, `scripts/check-charts.mjs:58`, `scripts/check-dist.mjs:22`, `scripts/check-dist.mjs:60`).
- JSDoc-emitted leaf declarations have a solid byte-compare loop: `tsconfig.dts.json` lists every JS input, `emit-dts` deletes old declarations before emit, and `check-dts-emit` re-emits to a temp dir and compares `.d.ts` plus maps (`tsconfig.dts.json:16`, `scripts/emit-dts.mjs:16`, `scripts/check-dts-emit.mjs:41`, `scripts/check-dts-emit.mjs:53`).
- Public package checks are derived from the manifest, not hand lists: export targets are flattened in one helper, pack checks require exported files in the tarball, and consumer type smoke imports every typed export through package resolution (`scripts/lib/package-targets.mjs:14`, `scripts/check-pack.mjs:74`, `scripts/check-consumer-types.mjs:49`).
- Common check plumbing is factored well: `writeGenerated`/`genBanner`, `freshnessErrors`, `reportAndExit`, shared CSS import/comment parsing, and renderer token reference resolution avoid repeated gate frames (`scripts/lib/emit.mjs:22`, `scripts/lib/assert-fresh.mjs:36`, `scripts/lib/gate-report.mjs:19`, `scripts/lib/patterns.mjs:9`, `scripts/lib/resolve-ref.mjs:1`).

## Weaknesses / risks
- [P1] `verify:artifacts` is not a complete artifact verifier: `build:artifacts` generates contrast and recipe-related declaration output, but `verify:artifacts` omits `check:contrast` and `check:recipe-types`; those only run in the full `check` chain, so `npm run prepack` can pass with semantic artifact drift that `npm run check` would catch (`package.json:149`, `package.json:187`, `package.json:200`, `package.json:201`, `package.json:202`).
- [P1] Recipe option types are still handwritten in `gen-dts.mjs`; `check-recipe-types` only covers string-literal options and explicitly skips numeric/boolean shapes, so drift like a missing boolean option or numeric range is structurally possible (`scripts/gen-dts.mjs:26`, `scripts/gen-dts.mjs:103`, `scripts/check-recipe-types.mjs:13`, `classes/index.js:941`, `classes/index.js:986`).
- [P2] `check-recipe-types` is regex- and formatting-dependent: it locates `const ui = {`, assumes two-space method headers, recognizes only two helper-map shapes, and scrapes `key === 'literal'`, so refactoring factories can create false negatives (`scripts/check-recipe-types.mjs:29`, `scripts/check-recipe-types.mjs:50`, `scripts/check-recipe-types.mjs:68`, `scripts/check-recipe-types.mjs:78`).
- [P2] The JSDoc declaration pipeline emits from comments but intentionally does not type-check JS implementations (`checkJs:false`, `noEmitOnError:false`, and `check-dts-emit` ignores `tsc` exit), so declaration freshness is guaranteed but implementation/type soundness is not (`tsconfig.dts.json:4`, `tsconfig.dts.json:14`, `scripts/check-dts-emit.mjs:41`, `scripts/check-dts-emit.mjs:49`).
- [P2] The artifact DAG is implicit in npm script order and module imports rather than declared metadata; adding one generated family requires updating build, verify, check, exports/files, and possibly dist leaf lists, with `check-chain` only guarding `check:*` membership (`package.json:200`, `package.json:201`, `scripts/check-chain.mjs:17`, `scripts/build-dist.mjs:86`, `package.json:252`).
- [P3] CSS leaf scalability is check-backed but still multi-registry: opt-in leaves live in `EXTRA_LEAVES`, package exports, and `files`; checks catch omissions, but adding a primitive remains a many-file operation (`scripts/build-dist.mjs:83`, `package.json:267`, `scripts/check-exports.mjs:77`, `scripts/check-dist.mjs:22`).

## Top recommendations
1. Replace `build:artifacts`/`verify:artifacts` with a declared artifact registry containing `sources`, `outputs`, `build`, and `checks`, then generate npm chains from it (`package.json:200`, `package.json:201`, `scripts/lib/artifacts.mjs:31`).
2. Add `check:contrast` and `check:recipe-types` to `verify:artifacts`, or rename that script to reflect that it is only byte-freshness plus selected semantic gates (`package.json:149`, `package.json:187`, `package.json:201`).
3. Move `ui.*` recipe option metadata to a structured source beside the runtime factories, and generate both runtime branches/types from it or validate the full object shape with AST parsing (`scripts/gen-dts.mjs:54`, `classes/index.js:909`, `scripts/check-recipe-types.mjs:13`).
4. Replace `check-recipe-types` regex scraping with `acorn`/TypeScript AST traversal or a small declarative recipe schema to remove formatting/helper-shape coupling (`scripts/check-recipe-types.mjs:45`, `scripts/check-recipe-types.mjs:68`, `scripts/check-recipe-types.mjs:78`).
5. Consider enabling `checkJs` for declaration-emitted JS leaves incrementally, or add a separate `tsc --allowJs --checkJs --noEmit` config for public signatures (`tsconfig.dts.json:4`, `scripts/check-dts-emit.mjs:8`).
6. Generate CSS export entries from `leafFiles()+EXTRA_LEAVES` or move leaf declarations into one manifest consumed by `build-dist`, `check-exports`, and `package.json` generation (`scripts/build-dist.mjs:78`, `scripts/build-dist.mjs:86`, `scripts/check-exports.mjs:100`).

## Notable observations
- The pipeline often re-derives from source instead of reading stale intermediates: Figma, charts, contrast, Mermaid/D2/Vega all call `buildResolved()` or `resolveRef` rather than trusting committed `tokens/resolved.json` (`scripts/gen-figma-variables.mjs:129`, `scripts/gen-charts.mjs:16`, `scripts/gen-contrast.mjs:263`, `scripts/lib/resolve-ref.mjs:17`).
- `check-chain` is a useful meta-gate for aggregate `check`, but it does not protect `build:artifacts` or `verify:artifacts`, which are the chains most exposed to generated-output omissions (`scripts/check-chain.mjs:1`, `scripts/check-chain.mjs:37`, `package.json:200`, `package.json:201`).
- The foreign renderer theme generators are architecturally consistent: each owns a local mapping, emits JS/JSON/DTS, and has a structural no-render gate for drift/coverage/resolvability (`scripts/gen-mermaid.mjs:8`, `scripts/check-mermaid.mjs:29`, `scripts/gen-d2.mjs:8`, `scripts/check-d2.mjs:27`, `scripts/gen-vega.mjs:9`, `scripts/check-vega.mjs:33`).
