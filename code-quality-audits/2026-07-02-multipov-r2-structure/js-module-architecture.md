# JS / behaviors module architecture — bronto-ui structure review
**Verdict:** The JS layer mostly matches the intended thin, optional contract: behaviors are documented as side-effect-free/SSR-safe (`docs/architecture.md:74`), current public initializers consistently guard DOM work before wiring (`behaviors/command.js:43`, `behaviors/connectors.js:95`, `behaviors/toast.js:153`), and package side effects are limited to CSS (`package.json:41`). The structure is acyclic and mostly one-way, but adapter entrypoints import the full behavior barrel (`react/index.js:49`) and class registry (`react/index.js:161`), so the new per-leaf behavior exports (`package.json:410`) are real for vanilla consumers but not fully reflected in framework adapter granularity.

**Grade:** B+ — clean core graph, but adapter/barrel granularity and duplicated adapter helpers hold it back.

## Strengths
- Behavior leaves are structurally granular for vanilla consumers: package exports map individual leaves (`package.json:410`, `package.json:490`), and leaves import shared internals directly instead of the barrel (`behaviors/dialog.js:1`, `behaviors/combobox.js:1`).
- Runtime import direction is clean: adapters sit above the behavior barrel (`react/index.js:49`, `svelte/index.js:36`), behavior leaves depend downward on `internal.js` (`behaviors/connectors.js:1`), connector DOM glue depends on the pure connector kernel (`behaviors/connectors.js:2`), and annotations depend on connectors (`annotations/index.js:204`).
- `internal.js` is a focused shared core rather than a random barrel: it owns no-op/DOM detection (`behaviors/internal.js:17`), delegation-host resolution (`behaviors/internal.js:50`), idempotent binding (`behaviors/internal.js:69`), scoped querying (`behaviors/internal.js:81`), host collection (`behaviors/internal.js:106`), and small interaction primitives (`behaviors/internal.js:114`, `behaviors/internal.js:130`, `behaviors/internal.js:144`).
- Import-time DOM safety is actively gated: packed public imports are checked for accidental DOM global creation (`scripts/check-consumer-surface.mjs:1`, `scripts/check-consumer-surface.mjs:290`), and public behavior no-DOM calls are tested across the barrel surface (`test/behaviors.test.mjs:82`).
- Connectors and annotations have a clear source boundary: connectors explicitly own pure geometry and no live tracking (`connectors/index.js:5`), the DOM behavior owns redraw/listener glue (`behaviors/connectors.js:80`), and annotations reuse the same geometry kernel (`annotations/index.js:1`, `annotations/index.js:204`).
- Public surface parity is guarded by derived matrices: behavior ownership is checked from `behaviors/index.js` (`scripts/check-behavior-matrix.mjs:165`), framework binding surfaces are derived from the barrel (`scripts/check-binding-matrix.mjs:60`), and helper exports are matrix-owned (`scripts/check-helper-matrix.mjs:22`).

## Weaknesses / risks
- [P1] Framework adapters are monolithic entrypoints: React imports every behavior from the barrel (`react/index.js:49`) and then exports wrappers for all behaviors (`react/index.js:108`), with the same pattern in Solid/Qwik/Svelte/Vue (`solid/index.js:51`, `qwik/index.js:55`, `svelte/index.js:36`, `vue/index.js:38`); cost: one adapter hook relies on downstream dead-code elimination and native/dev-server ESM import pulls the full static graph.
- [P1] Adapter subpaths are not behavior-only glue because all five re-export the class registry (`react/index.js:161`, `solid/index.js:165`, `qwik/index.js:222`, `svelte/index.js:193`, `vue/index.js:228`); cost: framework imports are coupled to the  class contract module in addition to lifecycle delegation (`classes/index.js:18`).
- [P2] The behavior barrel is curated but still full-inventory (`behaviors/index.js:33`); importing `@ponchia/ui/behaviors` pulls leaves such as glyph behavior (`behaviors/index.js:47`), which imports the glyph registry (`behaviors/glyph.js:2`, `glyphs/glyphs.js:31`); cost: vanilla consumers get real leaf imports, but barrel users pay parse/load cost for unrelated behavior families.
- [P2] Per-leaf JS exports are not protected by an inverse registry the way CSS leaves are: `check-exports` has an inverse CSS leaf export check (`scripts/check-exports.mjs:77`) and behavior matrix checks barrel ownership (`scripts/check-behavior-matrix.mjs:181`), but there is no shown gate requiring every `behaviors/*.js` leaf to have a `package.json` subpath; cost: future granular exports can drift silently.
- [P2] Shared adapter logic is duplicated rather than factored: React/Solid/Qwik each define resolver helpers (`react/index.js:75`, `solid/index.js:77`, `qwik/index.js:81`), while Svelte/Vue duplicate their own cleanup/root handling (`svelte/index.js:62`, `vue/index.js:64`); cost: lifecycle/root semantics changes need 5 edits, with tests asserting drift after the fact (`test/bindings.test.mjs:438`).
- [P3] `resolveHost` is convention-safe but not fail-closed by itself because its default parameter references `document` (`behaviors/internal.js:50`); current callers guard first (`behaviors/table.js:37`, `behaviors/table.js:38`), but cost is a future SSR foot-gun if a new helper or behavior calls it before `hasDom()`.
- [P3] Public declarations leak the internal module path: `internal.js` says it is not part of the public behaviors surface (`behaviors/internal.js:1`), but adapter `.d.ts` files reference `../behaviors/internal.js` (`react/index.d.ts:8`, `react/index.d.ts:39`); cost: internal file shape/name becomes part of the practical type contract.
- [P3] Connector low-level helpers are public because annotations need the kernel (`connectors/index.js:45`, `annotations/index.js:212`), and the helper matrix owns them as public helpers (`scripts/check-helper-matrix.mjs:60`); cost: numeric/kernel internals are harder to refactor without public API churn.

## Top recommendations
1. Split framework adapter imports away from `../behaviors/index.js` and import leaf modules directly for each wrapper (`react/index.js:49`, `package.json:410`).
2. Add a JS inverse export gate: every non-`index.js`/`internal.js` `behaviors/*.js` leaf must have matching package export/types entries (`scripts/check-exports.mjs:77`, `package.json:410`).
3. Extract or generate shared adapter resolver/root handling so React/Solid/Qwik/Svelte/Vue do not duplicate lifecycle semantics by hand (`react/index.js:75`, `svelte/index.js:66`, `vue/index.js:83`).
4. Make `resolveHost` SSR-safe independently of caller order, e.g. avoid `fallback = document` and fail closed when `globalThis.document` is absent (`behaviors/internal.js:50`).
5. Decide whether connector scalar helpers are intentionally public API or should move behind a private geometry kernel consumed by annotations (`connectors/index.js:45`, `scripts/check-helper-matrix.mjs:65`).
6. Factor repeated snapshot/restore helpers only if they can stay narrow; the duplication is visible across forms/table/legend/carousel (`behaviors/forms.js:3`, `behaviors/table.js:44`, `behaviors/legend.js:33`, `behaviors/carousel.js:11`).

## Notable observations
- The current per-leaf behavior exports are structurally real for vanilla imports, not cosmetic (`package.json:430`, `behaviors/dialog.js:1`).
- Svelte and Vue remain plain adapter objects with no Svelte/Vue runtime import, matching the optional no-peer intent (`svelte/index.js:36`, `vue/index.js:38`, `docs/architecture.md:91`).
- Some singleton runtime state is intentional but worth remembering: modal stack state lives at module scope (`behaviors/modal.js:24`) and toast queue state lives at module scope (`behaviors/toast.js:14`), but neither mutates DOM on import because DOM work starts inside exported functions (`behaviors/toast.js:153`).
