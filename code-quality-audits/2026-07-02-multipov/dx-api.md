# Developer experience & public API — bronto-ui review
**Verdict:** Adoption feels strong if the consumer accepts the CSS-first contract: root CSS is explicit, classes are generated/typed, behaviors are optional cleanup-returning glue, and docs are unusually honest about where CSS stops (`package.json:242`, `docs/reference.md:12`, `classes/index.js:1`, `behaviors/index.js:4`). Living with it still requires discipline: consumers author ARIA, `data-bronto-*`, imports, and contrast checks themselves, and a few sharp edges remain in types and adapter lifecycle behavior (`docs/usage.md:629`, `docs/theming.md:56`, `classes/index.d.ts:715`, `react/index.js:103`).
**Grade:** B+ — excellent contract machinery, held back by a few avoidable footguns (`docs/stability.md:57`, `classes/index.d.ts:715`).

## Strengths
- The class API is discoverable and auditable: `docs/reference.md` is generated, reports 644 classes / 179 groups, and points to `cls`, `classes.json`, and tokens as canonical surfaces (`docs/reference.md:1`, `docs/reference.md:12`, `docs/reference.md:13`).
- Naming is mostly coherent BEM-style: reference states "`--x` = modifier, `__x` = part", and `cls` maps literal keys to real `ui-*` selectors (`docs/reference.md:21`, `classes/index.js:18`).
- Recipes reduce typo risk without a component runtime: `ui.button`, `ui.table`, `ui.modal`, etc. return strings, and `attrs.meter/progress/dotbar` bundle ARIA plus `--value` so visual and announced values do not drift (`classes/index.js:902`, `classes/index.js:1071`).
- Type declarations are mostly high-value: class literals, option unions, token key unions, and `attrs` overloads give autocomplete and typo rejection (`classes/index.d.ts:666`, `classes/index.d.ts:901`, `tokens/index.d.ts:4`, `tokens/index.d.ts:26`).
- Override/theming story is technically sound: all default CSS is in `@layer bronto`, and root `--accent` rebrands buttons, focus rings, dot motifs, borders, and fills (`css/core.css:6`, `css/core.css:10`, `docs/theming.md:22`, `docs/theming.md:29`).
- Adapters are thin and consistent in intent: React/Solid/Qwik wrap behavior initializers as hooks, Svelte exposes actions, Vue exposes directives/plugin helpers, and all re-export `cls/ui/cx` or behavior glue without owning markup (`react/index.js:97`, `solid/index.js:99`, `qwik/index.js:106`, `svelte/index.js:121`, `vue/index.js:117`).
- Adoption and release contracts are unusually explicit: `files` allowlists shipped docs/data/adapters, `exports` enumerates CSS leaves and JS subpaths, stability defines pre-1.0 SemVer, and migrations are machine-readable (`package.json:44`, `package.json:242`, `docs/stability.md:3`, `MIGRATIONS.json:1`).

## Weaknesses / risks
- [P1] `ui.table({ breakAnywhere: true })` is implemented and documented, but `TableOpts` omits `breakAnywhere`; TypeScript consumers lose autocomplete and get rejected for a valid recipe option (`classes/index.js:933`, `classes/index.js:939`, `classes/index.d.ts:715`, `docs/reference.md:1392`).
- [P2] Behavior-required widgets can fail hard when the init is missed: docs say absent tabs can make content unreachable, controlled modals lack focus trap/Escape, menus lack dismissal, and validation markup silently no-ops when required attributes are missing (`docs/usage.md:629`, `docs/usage.md:636`, `docs/usage.md:643`, `docs/reference.md:1620`).
- [P2] Adapter lifecycle is not the same across frameworks: React/Solid/Qwik run once on mount/visible-task, while Svelte/Vue restart when action/directive values change; changing roots/options after mount behaves differently by framework (`react/index.js:103`, `solid/index.js:105`, `qwik/index.js:122`, `svelte/index.js:88`, `vue/index.js:107`).
- [P2] Import modes are well documented but easy to hold wrong: root import is CSS-only, package CSS specifiers do not work in plain HTML, and leaves switch from source `css/` names to built `dist/css/` URLs outside bundlers (`llms.txt:18`, `llms.txt:25`, `llms.txt:36`, `llms.txt:43`).
- [P2] The one-knob theme story is root-level, not arbitrary-subtree: overriding only `--accent` in a subtree leaves derived tokens stale, and custom accents make contrast the consumer’s responsibility (`docs/theming.md:32`, `docs/theming.md:40`, `docs/theming.md:56`, `docs/usage.md:670`).
- [P2] Runtime recipe guardrails are uneven: invalid tones warn, but other finite options are direct comparisons that silently drop invalid JS values; docs already call out density no-ops as a real trap (`classes/index.js:754`, `classes/index.js:760`, `classes/index.js:933`, `docs/usage.md:29`).
- [P3] “No `!important`” is true for normal component overrides but not literal: reduced-motion and print rules use `!important`, and the demo needs explicit layer ordering to beat the library’s important reduced-motion reset (`css/core.css:6`, `css/motion.css:396`, `css/base.css:258`, `demo/index.html:12`).
- [P3] `.ui-themetoggle` is a parts-only namespace with `base: null`; that preserves legacy shape but breaks the otherwise simple “base + parts/modifiers” mental model (`classes/classes.json:1384`, `classes/classes.json:1385`, `docs/reference.md:1444`).

## Top recommendations
1. Fix `TableOpts.breakAnywhere` and add a generator/type test that compares every `ui.*` option consumed in `classes/index.js` with the generated interfaces (`classes/index.js:933`, `classes/index.d.ts:715`).
2. Extend runtime dev warnings beyond tones to all closed recipe unions, especially density/variant/size/state values, because JS consumers do not get TypeScript’s guardrails (`classes/index.js:760`, `classes/index.js:902`, `docs/usage.md:61`).
3. Make adapter lifecycle semantics explicit and aligned: either document React/Solid/Qwik as one-shot bindings beside Svelte/Vue’s reactive actions, or add a dependency/rebind API for hook users (`react/index.d.ts:1`, `svelte/index.d.ts:106`, `vue/index.js:107`).
4. Add a “behavior-required markup checklist” or copy-paste snippets for tabs, menu, modal, combobox, validation, and disabled guard, since the no-component-runtime model otherwise leaves accessibility wiring to every adopter (`docs/usage.md:633`, `docs/usage.md:656`).
5. Put the import-mode matrix front and center in package docs: bundler CSS import, Node/runtime JS subpaths, plain HTML `dist/` links, CDN import maps, and opt-in leaf imports are all distinct (`docs/getting-started/vanilla.md:18`, `docs/getting-started/vanilla.md:22`, `llms.txt:43`).
6. Promote the theme playground and subtree rebrand recipe as first-class adoption tools, because “one `--accent`” is safe only at root/theme roots and contrast verification is mandatory for custom hues (`docs/theming.md:32`, `docs/theming.md:277`).

## Notable observations
- `@property --value` makes `--value: 50%` invalid and fall back to an empty bar instead of painting the wrong fill, which is a clever CSS-level guardrail (`css/feedback.css:7`, `css/feedback.css:13`).
- Behavior scoping distinguishes `root: undefined` from `root: null`, preventing an unready framework ref from accidentally wiring the whole document (`behaviors/internal.js:38`, `behaviors/internal.js:50`).
- `bindOnce` makes delegated behaviors idempotent, which protects HMR and React StrictMode-style double mounts from stacking duplicate listeners (`behaviors/internal.js:62`, `behaviors/internal.js:69`).
- The Tailwind bridge is CSS-only and intentionally does not import Bronto component CSS, so adopters must import both systems deliberately (`tailwind.css:1`, `tailwind.css:9`, `docs/interop/tailwind.md:28`).
- Static reports have a genuinely no-JS path through `dist/bronto.css` plus `report-kit.css`; the standalone demo documents pinned CDN/node_modules paths rather than package specifiers (`demo/report-standalone.html:3`, `demo/report-standalone.html:9`, `demo/report-standalone.html:26`).
