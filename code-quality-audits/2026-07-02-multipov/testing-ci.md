# Testing & CI quality gates — bronto-ui review
**Verdict:** The quality-gate story is mostly real for the publish path: `npm run check` is broad, meta-gated, and paired with Playwright, visual baselines, packed-consumer checks, contrast, type, and report grammar gates (`package.json:146`, `scripts/check-chain.mjs:37`, `.github/workflows/release.yml:55`, `.github/workflows/release.yml:95`). It is less absolute for PR merge quality: PR e2e is Chromium-only, cross-browser examples/visual smoke are deeper on manual/release paths, SAST is audit/advisory rather than a required PR gate, and a few areas are intentionally structural/advisory rather than pixel or axe-hard (`.github/workflows/ci.yml:131`, `.github/workflows/examples.yml:80`, `.github/workflows/scorecard.yml:3`, `test/e2e/report.spec.mjs:158`).
**Grade:** A- — unusually strong executable gates, with PR/release asymmetry and some advisory/manual coverage.

## Strengths
- The aggregate gate is concrete and self-policing: `check` chains lint, format, exports, generated freshness, knip, complexity, types, unit, dist, pack, consumer, matrices, examples, visual, schemas, workflows, contrast, color-policy, reports, and more (`package.json:146`), while `check-chain` fails if a new `check:*` script is not wired into the aggregate (`scripts/check-chain.mjs:1`, `scripts/check-chain.mjs:37`).
- Visual regression is real screenshot testing, not just narrative: Playwright uses deterministic screenshot settings and Chromium pixel baselines (`playwright.config.mjs:36`, `playwright.config.mjs:45`), the visual spec screenshots every `data-shot` section in dark/light plus RTL/dialog states (`test/e2e/visual.spec.mjs:8`, `test/e2e/visual.spec.mjs:51`, `test/e2e/visual.spec.mjs:64`), and a static inventory gate catches missing or orphaned PNG baselines (`scripts/check-visual-baselines.mjs:33`, `scripts/check-visual-baselines.mjs:90`).
- Accessibility and contrast are automated in multiple layers: axe scans block serious/critical WCAG and structural failures (`test/e2e/_demo-guards.mjs:29`, `test/e2e/_demo-guards.mjs:48`), demo axe runs cover both themes and RTL (`test/e2e/a11y.spec.mjs:33`, `test/e2e/a11y.spec.mjs:74`), and token/colorway WCAG floors fail the check gate (`scripts/check-contrast.mjs:38`, `scripts/check-contrast.mjs:89`, `test/skins.test.mjs:49`).
- Behavior coverage includes SSR/no-DOM, cleanup, and real browser ownership: the behavior matrix requires docs, unit, and browser proof for public behavior exports (`scripts/check-behavior-matrix.mjs:1`, `scripts/check-behavior-matrix.mjs:184`), unit tests cover no-DOM safety and cleanup (`test/behaviors.test.mjs:82`, `test/behaviors.test.mjs:1260`, `test/behaviors.test.mjs:2300`), and e2e tests exercise dialog, disclosure, theme, connectors, and listener cleanup paths (`test/e2e/behavior.spec.mjs:58`, `test/e2e/connectors.spec.mjs:106`).
- Adapter and consumer-surface gates are stronger than typical library smoke tests: binding parity is checked against behavior exports and docs/types ownership (`scripts/check-binding-matrix.mjs:1`, `scripts/check-binding-matrix.mjs:101`), React/Solid/Svelte/Vue lifecycle cleanup is unit-tested (`test/bindings.test.mjs:39`, `test/bindings.test.mjs:200`), and packed-tarball consumer checks import public JS/CSS/types through package exports (`scripts/check-consumer-surface.mjs:1`, `scripts/check-consumer-types.mjs:1`).
- CSS vocabulary and report grammar are machine-checked: class registry and stylesheet selectors are bidirectionally validated (`scripts/check-classes.mjs:1`, `scripts/check-classes.mjs:77`), recipe options are probed for runtime/type drift (`test/classes.test.mjs:256`, `scripts/check-recipe-types.mjs:122`), and report docs/snippets/ARIA/source cards/claims are structurally validated (`scripts/check-report.mjs:274`, `scripts/check-report.mjs:390`, `scripts/check-report.mjs:445`).

## Weaknesses / risks
- [P1] PRs do not get the same cross-engine browser gate as release: CI runs PR e2e with Chromium only (`.github/workflows/ci.yml:131`), while full cross-engine e2e is used on release (`.github/workflows/release.yml:95`); a Firefox/WebKit-only behavior or layout regression can merge to `main` before the stronger gate catches it.
- [P2] Security/static-analysis gates are not PR-blocking in the same way as `check`: Scorecard runs on push/schedule/manual, not pull requests (`.github/workflows/scorecard.yml:3`), uploads SARIF as an advisory scan (`.github/workflows/scorecard.yml:31`), and Semgrep/CodeQL/gitleaks/checkov appear as local audit artifacts rather than required workflow steps (`code-quality-audits/2026-06-12-ai-slop-pass/report.md:63`).
- [P2] Some contrast guarantees are intentionally WCAG-only: APCA failures warn but do not fail (`scripts/check-contrast.mjs:51`, `scripts/check-contrast.mjs:57`), and one modal axe scan disables `color-contrast` because of backdrop-filter sampling (`test/e2e/a11y.spec.mjs:54`); perceptual dark-mode or blended-surface regressions can pass if they still clear the existing hard checks.
- [P2] The theme playground is guard-only rather than axe/functional-deep tested: current e2e deliberately skips axe for it (`test/e2e/demos.spec.mjs:49`) and checks only console/pageerror/bad responses (`test/e2e/demos.spec.mjs:89`), so a broken swatch interaction, generated ratio display, or local a11y issue could escape.
- [P2] Ownership matrices can prove that a test file mentions a surface without proving assertion depth: the component matrix says it is a coverage map, not a behavioral test (`scripts/check-component-matrix.mjs:1`), ownership proof is based on file existence and needle inclusion (`scripts/lib/ownership-proof.mjs:38`), and at least one e2e assertion is vacuous (`test/e2e/marks.spec.mjs:31`).
- [P3] Visual fidelity is still Chromium-centered: screenshot baselines run under the Chromium project (`playwright.config.mjs:45`) with Firefox/WebKit limited to the non-pixel set (`playwright.config.mjs:17`), so browser-specific visual-only drift relies on structural/computed assertions rather than pixel diffs.
- [P3] Full report compositions are intentionally not pixel-snapshotted: report e2e uses structural and PDF assertions instead (`test/e2e/report.spec.mjs:158`, `test/e2e/report-print.spec.mjs:79`), which is good for stability but leaves some typography/spacing regressions outside visual baselines.
- [P3] Branch protection itself is claimed but not verifiable from repo files: docs say `main` requires `check` and `e2e` green (`CONTRIBUTING.md:43`), and CI provides a single `ci-gate` status intended for protection (`.github/workflows/ci.yml:144`), but the actual required-check setting is external GitHub state.

## Top recommendations
1. Run cross-engine e2e on PRs when engine-sensitive files change, reusing the existing diff logic instead of waiting for post-merge or release gates (`.github/workflows/ci.yml:66`, `.github/workflows/ci.yml:131`).
2. Add a required lightweight PR security job for Semgrep registry/security plus secret scanning, or explicitly relabel those audits as advisory rather than part of the CI quality gate (`code-quality-audits/2026-06-12-ai-slop-pass/report.md:63`, `.github/workflows/scorecard.yml:3`).
3. Promote selected APCA/dark-theme contrast checks from warnings to failures for small text and accent text, and add targeted non-false-positive checks for backdrop/blended modal surfaces (`scripts/check-contrast.mjs:51`, `test/e2e/a11y.spec.mjs:54`).
4. Replace mention-based ownership proof with assertion-oriented fixtures or required helper calls, and remove vacuous assertions such as the marks count check (`scripts/lib/ownership-proof.mjs:38`, `test/e2e/marks.spec.mjs:31`).
5. Add a small cross-browser visual/layout smoke for the highest-risk rendering surfaces: report pages, app shell, modals, tables, and chart primitives (`test/e2e/report.spec.mjs:158`, `test/e2e/report-primitives.spec.mjs:15`).
6. Codify or periodically verify GitHub branch protection so the intended `ci-gate` requirement is machine-checked, not just documented (`CONTRIBUTING.md:43`, `.github/workflows/ci.yml:144`).

## Notable observations
- The visual-baseline update workflow is unusually careful: it runs in the exact pinned Playwright container and explains that PRs opened with the default token will not trigger required checks and are intentionally unmergeable without `BASELINE_PAT` (`.github/workflows/visual-baselines.yml:8`, `.github/workflows/visual-baselines.yml:51`, `.github/workflows/visual-baselines.yml:132`).
- `ci-gate` is a deliberate aggregator to avoid branch-protection drift across matrix legs (`.github/workflows/ci.yml:144`), which is a good pattern if GitHub protection actually requires only that status.
- The report print/PDF tests are deeper than ordinary smoke tests: they parse generated PDFs and check page count, module text, and overprint behavior (`test/e2e/report-print.spec.mjs:79`, `test/e2e/report-print.spec.mjs:100`).
- `code-quality-audits/` is useful historical evidence, but it is archival/manual: findings are dated local passes (`code-quality-audits/2026-06-12-local-quality-pass/claims.json:7`) and the package gate explicitly forbids shipping audit directories (`scripts/check-pack.mjs:51`).
