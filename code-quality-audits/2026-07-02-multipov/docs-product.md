# Documentation, positioning & product — bronto-ui review
**Verdict:** `@ponchia/ui` has a coherent and differentiated thesis: CSS is the product, framework adapters are thin lifecycle glue, and the strongest lane is report/provenance/explanation tooling rather than generic component breadth (README.md:42, docs/architecture.md:24, docs/frontier-primitives.md:9). The docs are deep, accurate-looking, and unusually machine-checkable, but the public positioning is slightly split: README sells “services, tools, sites, and reports,” while ROADMAP says the real center is maintainer-owned agents and report tooling (README.md:11, ROADMAP.md:21).  
**Grade:** A- — excellent contract/docs machinery; target clarity and roadmap freshness hold it back (docs/package-contract.md:7, ROADMAP.md:10).
## Strengths
- Clear product boundary: plain class-based CSS is canonical, optional layers sit on top, and the root export is CSS-only with explicit JS subpaths (docs/architecture.md:24, docs/architecture.md:29, docs/architecture.md:102).
- Differentiation is real: it refuses to out-catalog Radix/shadcn-style kits and instead targets explanation, provenance, relationships, command access, workbench ergonomics, reports, and durable state (docs/frontier-primitives.md:3, docs/frontier-primitives.md:9).
- Adoption path is practical: README covers npm/CDN install, CSS import, class markup, typed recipes, behaviors, glyphs, theming, and framework examples (README.md:61, README.md:73, README.md:101, README.md:110, README.md:179).
- Contract quality is unusually high: generated package contract, shipped-doc allowlist, export matrix, drift gates, contrast gates, migrations, and packed-consumer checks are documented as release invariants (docs/package-contract.md:1, docs/architecture.md:163, docs/architecture.md:176).
- `llms.txt` is genuinely useful: it warns agents about CSS package specifiers vs plain HTML URLs, `dist/css/` paths, root import limits, explicit JS subpaths, and report leaf imports (llms.txt:18, llms.txt:36, llms.txt:50, llms.txt:219).
- Honest limitations are frequent: reports do not sanitize content, annotations do not own scales/DOM/editing, Vega is theming only, and generated-content UI avoids fake confidence widgets (docs/reporting.md:76, docs/annotations.md:13, docs/vega.md:18, docs/generated.md:89).
- Migration/stability story is credible: pre-1.0 minor breaks are explicit, public surfaces are classified, deprecation policy is documented, and `MIGRATIONS.json` separates safe rewrites from manual migrations (docs/stability.md:3, docs/stability.md:57, MIGRATIONS.json:1).
## Weaknesses / risks
- [P1] ROADMAP is stale relative to the package: it says “Last reconciled against `0.6.8`,” while package/changelog are at `0.6.10`; strategic docs lose trust when they lag release docs (ROADMAP.md:10, package.json:3, CHANGELOG.md:8).
- [P1] Target user is ambiguous: README pitches a broad public UI layer, but ROADMAP says it is maintained primarily for the maintainer’s agents/report tooling and not for marketing push; external adopters may misread support intent (README.md:11, ROADMAP.md:21, ROADMAP.md:246).
- [P2] The docs surface is huge and inventory-first: `docs/README.md` lists 57 docs across governance, app primitives, reports, renderers, adapters, ADRs, and migrations, but does not start with role-based paths for “app adopter,” “report author,” or “agent” (docs/README.md:11, docs/README.md:21, docs/README.md:33, docs/README.md:63).
- [P2] Examples are verification-first more than teaching-first: the examples README foregrounds packed-tarball CI proof and smoke coverage, while the demo calls itself both showcase and documentation; newcomers may get fixtures before a guided tutorial (examples/README.md:3, examples/README.md:16, demo/index.html:83).
- [P2] The “monochrome, one accent” tagline is catchy but technically underspecified: ADR-0001 says the README/package informal framing is superseded and the real model has status, display, and data-viz tiers; first-read positioning should say “rationed color,” not imply literal monochrome (README.md:15, docs/adr/0001-color-system.md:3, docs/adr/0001-color-system.md:65).
- [P2] Scope can feel sprawling despite the refusal list: docs cover identity, reports, sources, generated AI trust, state, workbench, command, Mermaid, D2, Vega, and Tailwind, while the strategy says only the report lane has a real consumer and command/workbench follow-ons are demand-gated (docs/README.md:26, docs/README.md:33, docs/README.md:63, docs/frontier-primitives.md:232).
- [P3] `llms.txt` slightly overclaims by saying generated declarations are “always-correct” and “never lie”; the underlying drift checks are strong, but semantics still live in prose docs and examples (llms.txt:9, llms.txt:11, docs/usage.md:61).
## Top recommendations
1. Reframe README’s opening around the actual primary user: “for maintainer-owned tools, agents, static reports, and service shells,” then list general public adoption as secondary (README.md:11, ROADMAP.md:21).
2. Reconcile ROADMAP to `0.6.10` or remove the version-specific reconciliation line; keep strategy docs as fresh as CHANGELOG/package metadata (ROADMAP.md:10, CHANGELOG.md:8, package.json:3).
3. Add a role-based “Start here” block above the docs inventory: app/service adopter, static report author, LLM/agent, theming/rebrand, framework adapter, contributor (docs/README.md:11, docs/README.md:63).
4. Promote the frontier thesis into README: explicitly compare against generic component kits, Tailwind interop, and renderer/theme targets so differentiation is visible before the docs deep dive (docs/frontier-primitives.md:3, docs/interop/tailwind.md:7, docs/vega.md:18).
5. Add small per-example READMEs or annotated walkthroughs; keep packed-tarball smoke value, but make examples teach the integration pattern, not just prove it (examples/README.md:3, examples/README.md:47).
6. Soften `llms.txt`’s “never lie” phrasing to “generated and drift-checked; prefer these over prose, then confirm behavior docs,” preserving trust without an absolute claim (llms.txt:9, llms.txt:11).
## Notable observations
- `llms.txt` is worth highlighting as product surface, not just metadata: it encodes agent-safe import rules, CDN/file-origin traps, and report-layer composition in one offline entrypoint (llms.txt:36, llms.txt:63, llms.txt:219).
- The theme playground is a real adoption instrument: it computes accent-family contrast and emits CSS/DTCG snippets for the “verify your hue” obligation (demo/theme-playground.html:56, demo/theme-playground.html:98, docs/theming.md:277).
- The strongest honest stance is “host owns state/data; Bronto owns grammar/geometry,” repeated across figures, annotations, command, workbench, and state docs (docs/figure.md:64, docs/command.md:20, docs/workbench.md:71, docs/state.md:15).
- The package ships docs as part of the product, including `llms.txt`, migrations, package contract, stability, theming, reporting, renderer docs, and ADRs (package.json:62, package.json:65, package.json:104, package.json:110).
