# Examples & demos as teaching — bronto-ui documentation review
**Verdict:** The demos are strong teaching raw material, especially `demo/index.html`, `demo/service.html`, and `demo/report-standalone.html`; the examples are broad and well-maintained but still read more like packed-tarball smoke fixtures than consumer learning paths (`examples/README.md`, `scripts/test-examples.mjs`). The IA gap is discoverability and bidirectional navigation: docs explain, demos show, examples prove, but learners have to stitch those surfaces together themselves (`docs/index.html`, `docs/README.md`, `README.md`).
**Grade:** B- — excellent coverage inventory and realistic demos, weak example pedagogy and doc/demo linking.
## Strengths
- Broad runnable stack coverage exists: vanilla, Astro, SvelteKit, Vue, React, Solid, Qwik, Tailwind, and static report are all inventoried and smoke-built from the packed tarball (`examples/README.md`, `scripts/lib/examples.mjs`, `scripts/check-examples.mjs`).
- React/Solid/Qwik/Svelte/Vue examples correctly exercise adapter subpaths, scoped roots/actions/directives, theme toggle, dialog, tabs, glyphs, and toast (`examples/react-vite/src/main.jsx`, `examples/solid-vite/src/main.jsx`, `examples/qwik-vite/src/main.jsx`, `examples/sveltekit/src/routes/+page.svelte`, `examples/vue-vite/src/App.vue`).
- The kitchen-sink demo is unusually useful as a visual reference: many analytical/report/tooling specimens include “Markup & recipe,” “Full specimen,” and “Docs” links (`demo/index.html`).
- Static report teaching is strong: the report docs explain bundler vs standalone CSS paths, report grammar, chart/annotation recipes, claim/evidence/source structure, and Chromium PDF export (`docs/reporting.md`, `demo/report-standalone.html`).
- The service-shell demo covers a realistic identity app shell with rail nav, topbar actions, filters, metrics, lifecycle state, and dense table markup (`demo/service.html`).
- Maintenance gates are strong: examples, bindings, CSS leaves, behaviors, and helpers all have ownership checks tying source, docs, demos, tests, and packed-example coverage together (`scripts/check-examples.mjs`, `scripts/check-binding-matrix.mjs`, `scripts/check-component-matrix.mjs`, `scripts/check-behavior-matrix.mjs`).
## Weaknesses / risks
- [P1] Examples are framed as verification artifacts first: the README leads with npm-pack/tarball CI mechanics, `test-examples` copies examples into temp dirs and installs the tarball, and `report-static` calls itself “Packed tarball smoke”; learner cost is reverse-engineering integration from fixture code instead of following a tutorial (`examples/README.md`, `scripts/test-examples.mjs`, `examples/report-static/index.html`).
- [P1] Docs do not make examples/demos first-class IA: `docs/index.html` routes only markdown docs, `docs/README.md` does not point “Getting started” readers to runnable examples, and framework guides omit direct “open this example” links; learner cost is losing the path from recipe to working project (`docs/index.html`, `docs/README.md`, `docs/getting-started/react-solid.md`, `docs/getting-started/sveltekit.md`, `docs/getting-started/vue.md`).
- [P1] The most common consumer task with no runnable framework example is a real service screen: app shell + form controls + filters + sortable table + modal/toast lifecycle. Those are shown in demos, but framework examples mostly stop at theme/toast/dialog/tabs/glyphs; learner cost is uncertainty about composing real app surfaces in React/Svelte/Vue/etc. (`demo/service.html`, `demo/index.html`, `examples/react-vite/src/main.jsx`, `examples/sveltekit/src/routes/+page.svelte`, `examples/vue-vite/src/App.vue`).
- [P1] Demo/doc linking is one-way and uneven: `demo/index.html` links many docs and full specimens, but standalone pages like service/workbench/command do not link back to docs, and leaf docs like workbench/command/state do not link their demo pages; learner cost is hitting a live specimen with no route to the explanatory recipe, or vice versa (`demo/index.html`, `demo/service.html`, `demo/workbench.html`, `demo/command.html`, `docs/workbench.md`, `docs/command.md`, `docs/state.md`).
- [P2] Framework parity is uneven: React/Solid/Qwik examples include extra chart/skin imports and richer adapter scoping, Svelte/Vue are close but thinner, while vanilla and Astro are minimal theme/toast or theme-only examples; learner cost is different stacks receiving different levels of confidence (`examples/react-vite/src/main.jsx`, `examples/qwik-vite/src/main.jsx`, `examples/sveltekit/src/routes/+page.svelte`, `examples/vue-vite/src/App.vue`, `examples/vanilla-vite/index.html`, `examples/astro/src/pages/index.astro`).
- [P2] The examples README overclaims teaching consistency: it says each example shows CSS loading, no-flash theme, and behavior lifecycle cleanup, but Astro has no explicit cleanup and several richer framework examples include hidden test controls that are not explained as teaching material; learner cost is mistaking smoke-only scaffolding for recommended app code (`examples/README.md`, `examples/astro/src/pages/index.astro`, `examples/react-vite/src/main.jsx`, `examples/vue-vite/src/App.vue`).
- [P2] Installed-package learners get docs but not examples/demos: `package.json` ships many docs but excludes `examples/` and `demo/`, and `examples/README.md` explicitly says examples are not published; learner cost is `node_modules/@ponchia/ui/docs` having recipes without local runnable companions (`package.json`, `examples/README.md`).
## Top recommendations
1. [safe-mechanical] Add example/demo links to every getting-started guide and to `docs/index.html`/`docs/README.md`: “Guide → runnable example → live demo/source” (`docs/getting-started/*.md`, `docs/index.html`, `docs/README.md`).
2. [needs-writing] Add per-example teaching notes or READMEs explaining import placement, no-flash theme, behavior lifecycle, root scoping, and what is test-only scaffolding (`examples/react-vite/src/main.jsx`, `examples/sveltekit/src/routes/+page.svelte`, `examples/README.md`).
3. [needs-writing] Add one realistic “service screen” example, then port the same task across key frameworks: app shell, filters/form, sortable table, dialog, toast, and cleanup (`demo/service.html`, `demo/index.html`, `docs/usage.md`).
4. [safe-mechanical] Make demo/doc links bidirectional and gate them: every `demo/*.html` header links its doc, every leaf doc links live demo + source (`demo/workbench.html`, `docs/workbench.md`, `scripts/check-doc-links.mjs`).
5. [needs-writing] Recast `examples/report-static` from “packed tarball smoke” into a teaching report scenario and explicitly contrast it with `demo/report-standalone.html` no-build usage (`examples/report-static/index.html`, `demo/report-standalone.html`, `docs/reporting.md`).
6. [needs-writing] Add a task matrix to `examples/README.md`: “theme setup,” “framework behavior adapter,” “service shell,” “forms/tables,” “static report,” “Tailwind bridge,” with paths to docs, examples, and demos (`examples/README.md`, `README.md`).
## Notable observations
- There is a productive tension: `docs/usage.md` says a kitchen-sink demo cannot teach decision rules, while `demo/index.html` says it is “the showcase and the documentation”; the demo is valuable, but it should be a visual companion, not the only bridge (`docs/usage.md`, `demo/index.html`).
- The automated matrices prove existence and drift control, not pedagogy or IA quality; a surface can have docs/demo/e2e ownership and still be hard for consumers to discover or learn from (`scripts/check-component-matrix.mjs`, `scripts/check-binding-matrix.mjs`).
- The strongest current teaching artifact is the report lane, not the framework lane: `docs/reporting.md` plus `demo/report-standalone.html` is much closer to an end-to-end consumer cookbook than any framework example (`docs/reporting.md`, `demo/report-standalone.html`, `examples/react-vite/src/main.jsx`).
