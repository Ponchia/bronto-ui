# Conceptual / explanatory docs — bronto-ui documentation review
**Verdict:** The docs teach the mental model well, but not in one canonical consumer-facing place. The README gives the thesis: CSS-first, zero runtime, rationed colour, `@layer bronto`, and primitive ownership (`README.md:11`, `README.md:21`, `README.md:58`). `architecture.md` explains why CSS is the universal layer and where Bronto stops (`docs/architecture.md:24`, `docs/architecture.md:52`, `docs/architecture.md:114`). `theming.md` and `contrast.md` now make the accent/skin/renderer/contrast story coherent (`docs/theming.md:7`, `docs/theming.md:198`, `docs/contrast.md:99`). The weakness is IA: readers must assemble “how Bronto thinks” from README, architecture, frontier, theming, ADRs, usage, and dots rather than being routed through a single explanatory spine (`docs/README.md:11`, `docs/frontier-primitives.md:14`, `docs/adr/0001-color-system.md:63`).

**Grade:** B+ - strong conceptual material, fragmented canonical home.

## Strengths
- README puts the main philosophy before install/how-to: CSS-first, no component runtime, governed colour tiers, resolved renderer theme data, `@layer bronto`, and primitive ownership are all stated up front (`README.md:11`, `README.md:21`, `README.md:53`, `README.md:58`, `README.md:69`).
- `architecture.md` is a solid rationale document: it explains why plain class CSS is canonical, why optional JS/bindings stay thin, why normal overrides work through one `@layer bronto`, and what Bronto refuses to own (`docs/architecture.md:24`, `docs/architecture.md:29`, `docs/architecture.md:52`, `docs/architecture.md:127`).
- The theming explanation is now mostly complete as a mental model, not just recipes: `--accent` derivation, root-scope limits, contrast obligations, `--button-text` vs `--on-accent`, full re-skins, root-only skins, chart palette, and external renderer themes are all explained (`docs/theming.md:7`, `docs/theming.md:39`, `docs/theming.md:63`, `docs/theming.md:79`, `docs/theming.md:116`, `docs/theming.md:198`, `docs/theming.md:234`).
- `contrast.md` clearly separates guaranteed framework floors from consumer rebrand obligations and explains why skins are gated but arbitrary rebrands are not (`docs/contrast.md:9`, `docs/contrast.md:14`, `docs/contrast.md:99`, `docs/contrast.md:237`).
- Stability/contract docs explain what consumers can rely on: pre-1.0 minor breaks, patch safety, public-surface matrix, 1.0 readiness, shipped-doc paths, and generated provenance (`docs/stability.md:3`, `docs/stability.md:14`, `docs/stability.md:57`, `docs/package-contract.md:7`, `docs/package-contract.md:301`).
- The consumer hubs reinforce the model in context: `usage.md` starts with “Color is rationed. Structure carries meaning,” while `reporting.md` explains report layers as opt-in document/analytical grammar rather than a hidden default bundle (`docs/usage.md:8`, `docs/reporting.md:25`, `docs/reporting.md:81`).

## Weaknesses / risks
- [P1] No single canonical “mental model” page exists; the core theses are split across README, architecture, frontier, theming, dots, and ADRs (`README.md:58`, `docs/architecture.md:52`, `docs/frontier-primitives.md:14`, `docs/theming.md:166`, `docs/dots.md:25`, `docs/adr/0001-color-system.md:63`). Comprehension cost: a new consumer can follow recipes without internalizing the governing model.
- [P1] `frontier-primitives.md` is both philosophy and historical/research log, yet docs route report readers there (`docs/README.md:15`, `docs/frontier-primitives.md:49`, `docs/frontier-primitives.md:101`, `docs/frontier-primitives.md:226`). Comprehension cost: readers must separate stable boundary doctrine from PR-era status notes and dormant-roadmap material.
- [P2] “Tier” vocabulary is overloaded: README and ADR use five colour tiers, `theming.md` says “Three additive” token tiers while listing four families including z-index, and dots calls display-expression “Tier-3” (`README.md:180`, `docs/adr/0001-color-system.md:65`, `docs/theming.md:166`, `docs/theming.md:189`, `docs/dots.md:25`). Comprehension cost: readers can confuse colour governance tiers with token convenience families.
- [P2] The cascade model is explained, but not isolated as a consumer concept; meanwhile `package-contract.md` exposes many stable `./css/unlayered/*` escape hatches with only “opt out of @layer bronto” wording (`README.md:60`, `docs/architecture.md:52`, `docs/package-contract.md:17`, `docs/package-contract.md:79`). Comprehension cost: consumers may bypass the intended override model instead of using normal unlayered app CSS.
- [P2] `package-contract.md` is accurate inventory, but weak as explanation of the unusual root-CSS/no-`src` layout; the rationale lives in `architecture.md` instead (`docs/package-contract.md:12`, `docs/package-contract.md:224`, `docs/architecture.md:158`). Comprehension cost: a reader opening the contract from README gets “what exists” before “why this shape exists.”
- [P3] Some conceptual cross-references are not concrete anchors, especially `contrast.md` pointing to theming “Re-brand obligations” when `theming.md` has no matching heading (`docs/contrast.md:239`, `docs/theming.md:63`, `docs/theming.md:292`). Comprehension cost: readers hunting the obligation have to scan prose blocks.

## Top recommendations
1. [needs-writing] Add `docs/mental-model.md` or `docs/concepts.md`: one canonical reader-facing page for CSS-first/no-runtime, one `@layer bronto`, colour tiers, primitive ownership, opt-in leaves, and package surface.
2. [needs-writing] Split `frontier-primitives.md` into a stable consumer “primitive boundary” explainer and a maintainer research/roadmap log.
3. [safe-mechanical] Fix `theming.md` “Token tiers”: rename to “Stable token families,” say four families, and explicitly distinguish it from ADR-0001’s five colour tiers.
4. [needs-writing] Add a “Cascade and overrides” section/page covering `@layer bronto`, import order, no normal `!important`, media-boundary exceptions, and when `css/unlayered/*` is appropriate.
5. [needs-writing] Add a short generated-or-handwritten preface to `package-contract.md` explaining why the package root is CSS-only and why published root directories are path-frozen.
6. [safe-mechanical] Repair conceptual anchors and glossary drift: add a real “Re-brand obligations” heading, normalize `skin`/`colorway` terms, and choose `color` vs `colour` consistently in docs.
